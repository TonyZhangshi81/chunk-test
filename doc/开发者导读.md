# 开发者导读

本文面向后续维护者，说明项目中上传、切块、检索三条主流程分别经过哪些模块，以及这些模块各自承担的职责。

## 总体入口

所有命令都从 [src/main.py](src/main.py) 进入，由 Click 定义 CLI 命令。

- `upload`：上传文件并保存文档元数据。
- `chunk`：按指定策略切块并写入向量。
- `search`：对指定策略执行检索、回答生成和评分。
- `compare`：对同一问题依次运行 RCTS、SC、JE 三种策略。

运行时依赖由 [src/main.py](src/main.py) 中的 `_build_runtime()` 统一创建，主要包括：

- 数据库会话：[src/models/database.py](src/models/database.py)
- 文档仓储：[src/repositories/document_repo.py](src/repositories/document_repo.py)
- Chunk 仓储：[src/repositories/chunk_repo.py](src/repositories/chunk_repo.py)
- 实验仓储：[src/repositories/experiment_repo.py](src/repositories/experiment_repo.py)
- Embedding 服务：[src/services/embedding_service.py](src/services/embedding_service.py)
- LLM 服务：[src/services/llm_service.py](src/services/llm_service.py)
- 对象存储服务：[src/services/storage_service.py](src/services/storage_service.py)

## 流程一：上传

命令入口：`python src/main.py upload --path <file>`

模块流转如下：

1. [src/main.py](src/main.py) 中的 `upload()` 作为入口，负责组装运行时、生成 `document_id`，并串联后续步骤。
2. [src/utils/file_parser.py](src/utils/file_parser.py) 中的 `parse_file()` 读取源文件：
   - `.txt` 直接按 UTF-8 解码。
   - `.docx` 通过 `python-docx` 解析段落并合并为正文。
3. [src/services/storage_service.py](src/services/storage_service.py) 中的 `upload_bytes()` 把原始文件字节上传到 MinIO。
4. [src/models/document.py](src/models/document.py) 定义文档表结构；[src/main.py](src/main.py) 负责构造 `Document` 实例。
5. [src/repositories/document_repo.py](src/repositories/document_repo.py) 中的 `create()` 把文档元数据和正文写入数据库。

这一流程的产出有两部分：

- MinIO 中的一份原始文件。
- PostgreSQL 中的一条 `t_document` 记录。

## 流程二：切块

命令入口：`python src/main.py chunk --doc-id <uuid> --type <RCTS|SC|JE>`

模块流转如下：

1. [src/main.py](src/main.py) 中的 `chunk()` 调用 `_chunk_document()`。
2. `_require_document()` 先通过 [src/repositories/document_repo.py](src/repositories/document_repo.py) 读取文档正文。
3. [src/services/chunk_strategies/__init__.py](src/services/chunk_strategies/__init__.py) 中的 `build_strategy()` 根据策略类型创建具体策略对象。

### RCTS 路径

- 实现文件：[src/services/chunk_strategies/rcts_strategy.py](src/services/chunk_strategies/rcts_strategy.py)
- 使用 `RecursiveCharacterTextSplitter` 按固定窗口大小与 overlap 切分。
- 切分结果再交给 [src/services/chunk_strategies/payloads.py](src/services/chunk_strategies/payloads.py) 计算 `chunk_index`、`start_pos`、`end_pos`。
- 向量由 [src/services/embedding_service.py](src/services/embedding_service.py) 统一生成。

### SC 路径

- 实现文件：[src/services/chunk_strategies/sc_strategy.py](src/services/chunk_strategies/sc_strategy.py)
- 优先使用 LangChain 的 `SemanticChunker`。
- 若库不可用，则退回本地语义切块：
  - 先按正则切句。
  - 再调用 [src/services/embedding_service.py](src/services/embedding_service.py) 为每句生成向量。
  - 利用相邻句向量余弦相似度和分位数阈值判断断点。
- 最终仍通过 [src/services/chunk_strategies/payloads.py](src/services/chunk_strategies/payloads.py) 统一补齐位置信息。

### JE 路径

- 实现文件：[src/services/chunk_strategies/je_strategy.py](src/services/chunk_strategies/je_strategy.py)
- 先做本地粗粒度预切分，再调用 Jina Embeddings API。
- Jina 返回 chunk 向量后，策略会直接把向量附着到 payload 上。
- 因为 payload 已经带有 `embedding`，主流程不会再次走共享 embedding 服务。

### 持久化阶段

无论哪条策略路径，最终都会回到 [src/main.py](src/main.py) 中的 `_chunk_document()`：

1. 调用 [src/models/chunk.py](src/models/chunk.py) 中的 `build_embedding_column_values()`，把向量写到正确维度的列。
2. 构造 `Chunk` ORM 对象。
3. 通过 [src/repositories/chunk_repo.py](src/repositories/chunk_repo.py) 中的 `replace_for_document()` 整批替换该文档该策略下的 chunk 结果。

这一流程的产出是 `t_chunk` 表中的一组 chunk 记录。

## 流程三：检索

命令入口：`python src/main.py search --doc-id <uuid> --type <RCTS|SC|JE> --query <question>`

模块流转如下：

1. [src/main.py](src/main.py) 中的 `search()` 调用 `_search_document()`。
2. `_search_document()` 再次通过 [src/services/chunk_strategies/__init__.py](src/services/chunk_strategies/__init__.py) 创建策略对象。
3. 查询向量生成分两种情况：
   - RCTS、SC：通过 [src/services/embedding_service.py](src/services/embedding_service.py) 生成。
   - JE：通过 [src/services/chunk_strategies/je_strategy.py](src/services/chunk_strategies/je_strategy.py) 中的 `embed_query()` 调用 Jina 生成。
4. [src/repositories/chunk_repo.py](src/repositories/chunk_repo.py) 中的 `search_similar()`：
   - 按 `document_id`、`chunk_type`、`embedding_model` 过滤。
   - 根据查询向量维度动态选择向量列。
   - 用 pgvector 的 `cosine_distance` 做排序。
5. 取回的 chunk 文本作为上下文传给 [src/services/llm_service.py](src/services/llm_service.py) 中的 `answer()` 生成回答。
6. [src/services/quality_evaluator.py](src/services/quality_evaluator.py) 中的 `evaluate()` 使用 embedding 相似度给本次回答打分。
7. [src/models/experiment.py](src/models/experiment.py) 定义实验表结构；[src/repositories/experiment_repo.py](src/repositories/experiment_repo.py) 把本次问答记录写入 `t_experiment`。

这一流程的产出有两部分：

- CLI 输出的 JSON 回答结果。
- 数据库中的一条实验记录，供后续比较与回溯。

## 配置与基础设施

- [src/config.py](src/config.py)：统一读取环境变量配置。
- [src/logging_config.py](src/logging_config.py)：统一初始化日志格式与级别。
- [src/models/database.py](src/models/database.py)：管理 SQLAlchemy 引擎、会话和建表逻辑。
- [src/.env.example](src/.env.example)：本地配置模板。

## 维护建议

1. 若新增切块策略，优先遵守 [src/services/chunk_strategies/base.py](src/services/chunk_strategies/base.py) 的统一接口，再在 [src/services/chunk_strategies/__init__.py](src/services/chunk_strategies/__init__.py) 中注册。
2. 若新增 embedding 模型维度，需同步修改 [src/models/chunk.py](src/models/chunk.py) 中的维度映射，并确认数据库表结构支持该维度。
3. 若调整检索评估逻辑，应同时检查 [src/services/quality_evaluator.py](src/services/quality_evaluator.py) 和实验落库字段，避免对比结果失真。
